# AIS Demo — Reference Documents

**Project:** `ais_demo`

Every standard, guideline, and technical reference consulted during
the design and implementation of this project. Each entry describes
what the document contains, which sections are most relevant, and
where to obtain it.

---

## 1. IMO — AIS Overview Page

**Title:** Automatic Identification Systems (AIS)  
**Publisher:** International Maritime Organization  
**URL:** https://www.imo.org/en/ourwork/safety/pages/ais.aspx  
**Access:** Free  

**What it is:**  
The IMO's official summary page for AIS. Provides the regulatory
context — which ships must carry AIS under SOLAS V/19, the phased
implementation timeline (2002–2008), and links to the underlying
resolutions. Not a technical specification but a good starting point
for understanding the regulatory landscape.

**Most relevant to this project:**  
Understanding why AIS is mandatory, which vessel classes are required
to carry it, and the distinction between Class A (full SOLAS
requirement) and Class B (voluntary/non-SOLAS). Confirms that
the project's target — Class A shipborne equipment — is the
primary standard.

## 2. IMO Resolution A.1106(29) — Operational Use Guidelines

**Title:** Revised Guidelines for the Onboard Operational Use of
Shipborne Automatic Identification Systems (AIS)  
**Publisher:** International Maritime Organization  
**Adopted:** 2 December 2015 (29th IMO Assembly session)  
**URL:** https://wwwcdn.imo.org/localresources/en/OurWork/Safety/Documents/AIS/Resolution%20A.1106(29).pdf  
**Access:** Free PDF  
**Supersedes:** Resolution A.917(22) as amended by A.956(23)

**What it is:**  
The operational guidelines for ship officers using AIS. Written from
the bridge perspective: how to operate the transceiver, what data
to manually enter at voyage start, how to interpret received data,
and the limitations of AIS in collision avoidance.

**Document structure:**
- Purpose and cautions (§1–§3)
- AIS system description with component diagram (§4–§9)
- Ship data content — Table 1: static, dynamic, and voyage-related
  fields with update rates (§12–§14)
- Short safety-related messages (§15–§20)
- Onboard operation procedures (§22–§27)
- Inherent limitations (§32–§39)
- Collision avoidance use (§40–§44)
- VTS, SAR, and AtoN applications (§45–§52)
- Appendix 1: AIS component diagram (CPU, VHF-Tx, VHF-Rx×2,
  VHF-Rx ch70, GNSS-Rx, heading sensor, speed sensor, ROT sensor)
- Appendix 2: Technical description — VHF channels (161.975 MHz
  and 162.025 MHz), TDMA time slots (4500/min), overload behaviour

**Most relevant sections for this project:**

- **Table 1 (§13)** — Complete field-by-field breakdown of static,
  dynamic, and voyage-related data with update frequencies. Confirms
  which fields are auto-generated by sensors vs manually entered by
  the officer.
- **Table 2 (§14)** — Updated Class A reporting intervals (the
  authoritative version for the clean implementation, slightly
  different from the older MSC.74(69) table — notably anchored
  vessels split into ≤3 knots and >3 knots).
- **Table 3 (§14)** — Class B reporting intervals for reference.
- **Appendix 2** — Confirms the two AIS VHF frequencies and
  (S)TDMA channel access scheme.

**Key difference from MSC.74(69):**  
A.1106(29) Table 2 adds a new row: "at anchor or moored moving
faster than 3 knots → 10 seconds." The prototype used only the
MSC.74(69) table which had a single anchor row at 3 minutes. The
clean implementation should use the more detailed A.1106(29) table.

## 3. IMO Resolution MSC.74(69) — AIS Performance Standards

**Title:** Adoption of New and Amended Performance Standards —
Annex 3: Recommendation on Performance Standards for an Universal
Shipborne Automatic Identification System (AIS)  
**Publisher:** IMO Maritime Safety Committee  
**Adopted:** 12 May 1998  
**URL:** https://wwwcdn.imo.org/localresources/en/OurWork/Safety/Documents/AIS/Resolution%20MSC.74(69).pdf  
**Access:** Free PDF

**What it is:**  
The foundational IMO performance standard for AIS. Defines what a
Class A shipborne AIS must be capable of — not how it works at
the bit level (that is ITU-R M.1371), but what functional
requirements it must meet. Written for equipment manufacturers and
type-approval bodies.

**Document structure (Annex 3):**
- §1 Scope
- §2 Functionality — autonomous, assigned, and polling modes
- §3 Capability — communication processor, GNSS interface,
  sensor interfaces, manual input/output, error checking, BITE
- §4 User interface
- §5 Identification — MMSI
- §6 Information — Static/Dynamic/Voyage-related fields and
  update rates; **Table 1** (the original reporting rate table)
- §7 Permissible initialisation period (2 minutes)
- §8 Power supply
- §9 Technical characteristics

**Most relevant sections for this project:**

- **§6.1** — The authoritative list of AIS data fields divided into
  Static (MMSI, call sign, name, type, dimensions), Dynamic
  (position, time, COG, SOG, heading, nav status, ROT), Voyage
  (draught, hazardous cargo, destination, ETA), and Safety messages.
  This list is the source of the `AISMsg1` struct field set.

- **Table 1 (§6.2)** — The original reporting rate table used in
  the prototype:

  | Ship condition | Interval |
  |---|---|
  | At anchor | 3 min |
  | 0–14 kts | 12 s |
  | 0–14 kts, changing course | 4 s |
  | 14–23 kts | 6 s |
  | 14–23 kts, changing course | 2 s |
  | > 23 kts | 3 s |
  | > 23 kts, changing course | 2 s |

  The clean implementation should prefer A.1106(29) Table 2 which
  is more recent, but this table is the original normative source.

- **§6.2 update rates** — Confirms static data every 6 minutes,
  dynamic data per Table 1, voyage data every 6 minutes.

## 4. IMO SN/Circ.227 — AIS Installation Guidelines

**Title:** Guidelines for the Installation of a Shipborne Automatic
Identification System (AIS)  
**Publisher:** IMO Sub-Committee on Safety of Navigation (NAV)  
**Issued:** 6 January 2003  
**URL:** https://wwwcdn.imo.org/localresources/en/OurWork/Safety/Documents/AIS/SN.1-Circ.227.pdf  
**Access:** Free PDF

**What it is:**  
Practical installation guidelines for shipyard engineers, AIS
equipment installers, and marine surveyors. Covers antenna placement,
cabling, power supply, sensor connections, and static data entry.
Not a protocol specification, but directly relevant to understanding
what physical inputs feed an AIS transponder.

**Document structure:**
- §1 General — survey and documentation requirements
- §2 AIS Installation — VHF antenna, GNSS antenna, power, sync
- §3 Bridge Arrangement — minimum keyboard/display, pilot plug,
  display system, BIIT alarm
- §4 Dynamic data input — external sensors (IEC 61162-1/2),
  position/COG/SOG, heading, ROT, navigational status
- §5 Static Information — MMSI, IMO number, call sign, ship name,
  type, dimensions (A/B/C/D reference point diagram)
- §6 Long-range function
- Annex 1: Rate of Turn sensor input overview with the ROT
  encoding formula (ROTAIS = 4.733 × √ROTsensor)
- Annex 2: Ship type table (ITU-R M.1371 Table 18)
- Annex 3: Recommended IEC 61162 NMEA sentences

**Most relevant sections for this project:**

- **§4 and Annex 3** — The NMEA sentence types that feed an AIS
  transponder: `GNS`/`GGA`/`RMC` for position, `VBW`/`VTG` for
  SOG, `HDT` for heading, `ROT` for rate of turn. These are the
  sentence formatters the demo's sensor simulation must produce.

- **Annex 1** — The ROT encoding formula. ROT data is stored in the
  `AISMsg1.rot` field as a non-linear int8_t. This annex explains
  exactly how to convert between the sensor value (°/min) and the
  encoded ROTAIS value. Relevant when implementing ROT support in
  the clean version.

- **Annex 2** — The ship type coding table (two-digit identifiers,
  e.g. 70 = cargo, 80 = tanker, 60 = passenger). Used in
  `ship_type_name()` in vessel.c and the HTML map colour coding.

- **§5.2–§5.3** — The A/B/C/D reference point diagram for ship
  dimensions. Relevant if implementing the ship dimensions fields
  from Type 5 messages.

## 5. ITU-R M.1371-6 — AIS Technical Standard

**Title:** Technical Characteristics for an Automatic Identification
System Using Time Division Multiple Access in the VHF Maritime
Mobile Frequency Band  
**Publisher:** International Telecommunication Union — Radiocommunication Sector  
**Latest version:** M.1371-6 (February 2026)  
**URL:** https://www.itu.int/dms_pubrec/itu-r/rec/m/R-REC-M.1371-6-202602-I!!PDF-E.pdf  
**Access:** Free PDF (ITU account required)  
**Also available at:** https://www.itu.int/rec/R-REC-M.1371/en

**What it is:**  
The definitive technical specification for AIS at the protocol level.
Everything in `ais.c` is derived from this document. Defines the
exact bit layout of all 27 AIS message types, the 6-bit ASCII
payload encoding, the TDMA channel access scheme, operating
frequencies, modulation, and error correction.

**Versions:**
- M.1371-1 (2001) — original specification
- M.1371-4 (2010) — became free to download, added messages 25–27
- M.1371-5 (2014) — added Type 27 long-range message
- M.1371-6 (2026) — current version

**Most relevant sections for this project:**

- **Table 46 — Message Type 1 bit layout** — The primary source for
  every field offset and width in the `AISMsg1` struct and the
  `ais_encode_msg1` / `ais_parse_nmea` functions:
  - Bits 0–5: Message type (6 bits)
  - Bits 8–37: MMSI (30 bits)
  - Bits 38–41: Navigational status (4 bits)
  - Bits 42–49: Rate of Turn (8 bits, signed)
  - Bits 50–59: Speed Over Ground (10 bits, 1/10 knot)
  - Bits 61–88: Longitude (28 bits signed, 1/10000 min)
  - Bits 89–115: Latitude (27 bits signed, 1/10000 min)
  - Bits 116–127: Course Over Ground (12 bits, 1/10 degree)
  - Bits 128–136: True heading (9 bits)
  - Bits 137–142: Time stamp (6 bits)
  - Bits 148: RAIM (1 bit)

- **§3.3.3 — Payload character encoding** — Defines the 6-bit ASCII
  codec used in `encode_6bit()` and `decode_6bit()`.

- **Table 47 — Message Type 5 bit layout** — The source for Phase 2
  (static/voyage data): ship name (120 bits of 6-bit ASCII), IMO
  number (30 bits), call sign (42 bits), destination (120 bits),
  ETA (20 bits). Type 5 is 426 bits, always transmitted in two
  NMEA fragments.

- **§3.3.1 — TDMA frame structure** — Explains the 4500 time slots
  per minute, and why the system can handle 2000 vessels with
  100% throughput within 8–10 nm.

- **Table 18 — Navigational status codes** — Source of the
  `ais_nav_status_str[]` array (values 0–15).

- **Table 18 — Ship type codes** — Source of the Annex 2 table in
  SN/Circ.227, used in `ship_type_name()`.

**Note on version used:**  
The prototype was developed against M.1371-5. The bit layouts for
Types 1–3 are identical in M.1371-6. Any differences in M.1371-6
affect newer message types not yet implemented.

## 6. NMEA — Official Standard

**Title:** NMEA 0183 Standard for Interfacing Marine Electronic Devices  
**Publisher:** National Marine Electronics Association  
**URL:** https://www.nmea.org/  
**Access:** Paid (available from the NMEA store)

**What it is:**  
The formal specification for the NMEA 0183 sentence format: the
framing structure (`!` or `$` prefix, comma-delimited fields, `*XX`
checksum suffix), the XOR checksum algorithm, field encoding rules,
and all standard sentence types. The `!AIVDM` / `!AIVDO` sentence
format used throughout this project is defined within this standard
(via harmonisation with IEC-PAS 61162-100).

**Most relevant to this project:**  
The checksum algorithm (`nmea_cs()` in ais.c), field delimiter rules
(commas, empty fields allowed), and the sentence framing convention.

**Practical note:**  
The NMEA standard is expensive and legally encumbered. For AIS
software development, the Eric Raymond AIVDM guide (Reference 7)
is a complete practical substitute for the AIS-relevant portions.

## 7. AIVDM/AIVDO Protocol Decoding Guide

**Title:** AIVDM/AIVDO Protocol Decoding  
**Author:** Eric S. Raymond (esr@thyrsus.com)  
**Version:** 1.58 (24 June 2023)  
**URL:** https://gpsd.gitlab.io/gpsd/AIVDM.html  
**Access:** Free  
**Part of:** The GPSD project (https://gpsd.gitlab.io/gpsd/)

**What it is:**  
The most practically useful free reference for AIS software
development. Assembles information from ITU-R M.1371, IALA Technical
Clarifications, US Coast Guard AIS documentation, and real-world
implementation experience into a single readable guide. Covers all
27 AIS message types with complete field tables, worked decode
examples, and notes on real-world behaviour that the official
specifications omit.

**Document structure:**
- AIVDM/AIVDO sentence layer — framing, talker IDs, multi-fragment
- Payload armoring — the 6-bit ASCII codec with the full character
  mapping table
- AIS payload data types — u/U (unsigned), i (signed), x (spare),
  t (text), b (boolean) field type notation
- Types 1–3: Position Report Class A — complete field tables with
  all sentinel values documented
- Type 4: Base Station Report
- Type 5: Static and Voyage Related Data — two-fragment encoding
- Types 6–27: all remaining message types
- Byte alignment and bit stuffing — fill bits explained
- JSON-AIS encoding
- AIS feed sites (AISHub, MarineTraffic, etc.)

**Most relevant sections for this project:**

- **AIVDM/AIVDO Payload Armoring** — The most accessible description
  of the 6-bit codec, with the complete character-to-value mapping
  table. Confirms the `encode_6bit()` / `decode_6bit()` formulas.

- **Types 1, 2, and 3: Position Report Class A** — Full field table
  with all bit offsets, widths, types, and sentinel values. This is
  the reference for Suite N (real-world test sentences) — the page
  provides decoded values for example sentences.

- **Type 5: Static and Voyage Related Data** — Essential reading
  before implementing Phase 2. Documents the two-fragment structure,
  6-bit text encoding for ship name and destination, and the ETA
  encoding (month/day/hour/minute packed into 20 bits).

- **AIS Payload Byte Alignment, Padding, and Bit Stuffing** —
  Explains fill bits (the last field in the NMEA sentence before
  the checksum). The fill bits indicate how many bits at the end
  of the last payload character should be ignored. The prototype
  accepted the fill bits field but did not act on it — the clean
  implementation must handle this correctly for Type 5.

- **AIS feed sites** — Lists live data sources for Phase 3
  (real-time input): AISHub (free pool with raw NMEA feed),
  MarineTraffic (subscription), and others.

**Why this is the most important reference to keep open:**  
The ITU-R M.1371 specification is the normative standard, but it is
dense, formally structured, and expensive before M.1371-4 became
free. The Raymond guide covers the same technical ground in a format
optimised for developers. Known real-world quirks (e.g., some vessels
transmit heading as 0 instead of 511 when no heading sensor is
fitted; some AIS encoders produce COG values slightly above 359.9°)
are documented here but not in the official standards.
